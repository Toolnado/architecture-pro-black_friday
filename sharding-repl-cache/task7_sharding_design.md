## Онлайн-магазин «Мобильный мир»

## 1. Коллекция `orders` (Заказы)

### Схема документа

```json
{
  "_id": ObjectId("..."),
  "customer_id": "usr_12345",
  "created_at": ISODate("2024-11-20T14:32:00Z"),
  "items": [
    {
      "product_id": "prod_789",
      "name": "Смартфон X",
      "quantity": 1,
      "price": 49990.00
    },
    {
      "product_id": "prod_321",
      "name": "Чехол для смартфона",
      "quantity": 2,
      "price": 990.00
    }
  ],
  "status": "delivered",
  "total_amount": 51970.00,
  "geo_zone": "RU-MOW"
}
```

### Индексы

```js
// Поиск истории заказов пользователя (основной запрос)
db.orders.createIndex({ customer_id: 1, created_at: -1 })

// Отображение статуса заказа
db.orders.createIndex({ _id: 1, status: 1 })
```

### Выбор шард-ключа и стратегии

**Шард-ключ:** `{ customer_id: "hashed" }`

**Стратегия:** хешированное шардирование

**Обоснование:**

| Критерий | Оценка |
|---|---|
| Кардинальность | Высокая - миллионы уникальных пользователей |
| Равномерность | Хеш даёт равномерное распределение без хотспотов |
| Целевые запросы | История заказов `{ customer_id }` - все данные пользователя на одном шарде |
| Запись | Новые заказы распределяются по всем шардам, нет монотонного роста |

**Почему не другие варианты:**
- `_id` hashed - запросы по `customer_id` становятся scatter-gather по всем шардам
- `geo_zone` - низкая кардинальность (~10 зон), Москва создаёт хотспот
- `created_at` range - все новые заказы пишутся только в один шард

### Команды MongoDB

```javascript
// Включить шардирование базы данных
sh.enableSharding("somedb")

// Создать хешированный индекс перед шардированием
db.orders.createIndex({ customer_id: "hashed" })

// Зашардировать коллекцию
sh.shardCollection("somedb.orders", { customer_id: "hashed" })
```

## 2. Коллекция `products` (Товары)

### Схема документа

```json
{
  "_id": ObjectId("..."),
  "product_id": "prod_789",
  "name": "Смартфон X",
  "category": "electronics",
  "price": 49990.00,
  "stock": {
    "RU-MOW": 50,
    "RU-EKB": 50,
    "RU-KGD": 30
  },
  "attributes": {
    "color": "black",
    "storage_gb": 128,
    "brand": "BrandName"
  }
}
```

### Индексы

```javascript
// Поиск по категории + фильтр по цене
db.products.createIndex({ category: 1, price: 1 })

// Страница товара
db.products.createIndex({ product_id: 1 }, { unique: true })
```

### Выбор шард-ключа и стратегии

**Шард-ключ:** `{ category: 1, _id: 1 }`

**Стратегия:** диапазонное (range) шардирование

**Обоснование:**

| Критерий | Оценка |
|---|---|
| Запросы по категории | Попадают на 1–2 шарда, а не scatter-gather |
| Фильтр по цене | После попадания на шард по category - эффективный range scan по индексу `{ category, price }` |
| Равномерность | `_id` как второй компонент разбивает данные внутри категории равномерно |
| Обновление остатков | Точечное обновление по `product_id` с известной категорией - targeted write |

**Почему не другие варианты:**
- `category` один - низкая кардинальность (~10–20 категорий), неравномерное распределение, «Электроника» создаёт хотспот
- `_id` hashed - поиск по категориям становится scatter-gather по всем шардам
- `price` - частые обновления цен создают постоянный data migration между чанками

### Команды MongoDB

```javascript
// Создать составной индекс
db.products.createIndex({ category: 1, _id: 1 })

// Зашардировать коллекцию
sh.shardCollection("somedb.products", { category: 1, _id: 1 })
```

## 3. Коллекция `carts` (Корзины)

### Схема документа

```json
{
  "_id": ObjectId("..."),
  "user_id": "usr_12345",
  "session_id": null,
  "items": [
    {
      "product_id": "prod_789",
      "quantity": 1
    },
    {
      "product_id": "prod_321",
      "quantity": 2
    }
  ],
  "status": "active",
  "created_at": ISODate("2024-11-20T14:00:00Z"),
  "updated_at": ISODate("2024-11-20T14:30:00Z"),
  "expires_at": ISODate("2024-11-27T14:00:00Z")
}
```

Гостевая корзина - `user_id: null`, заполнен `session_id`:

```json
{
  "_id": ObjectId("..."),
  "user_id": null,
  "session_id": "sess_abc123xyz",
  "items": [...],
  "status": "active",
  "created_at": ISODate("2024-11-20T14:00:00Z"),
  "updated_at": ISODate("2024-11-20T14:05:00Z"),
  "expires_at": ISODate("2024-11-21T14:00:00Z")
}
```

### Индексы

```javascript
// Получение активной корзины пользователя
db.carts.createIndex({ user_id: 1, status: 1 })

// Получение активной гостевой корзины
db.carts.createIndex({ session_id: 1, status: 1 })

// TTL-индекс для автоматической очистки
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
```

### Выбор шард-ключа и стратегии

**Шард-ключ:** `{ _id: "hashed" }`

**Стратегия:** хешированное шардирование

**Обоснование:**

| Критерий | Оценка |
|---|---|
| Кардинальность | Высокая - каждый `_id` уникален |
| Равномерность | Хеш обеспечивает равномерное распределение корзин |
| Гостевые корзины | Нет проблемы с `user_id: null` - ключом является `_id` |
| Объём данных | Активных корзин немного; scatter-gather по `user_id`/`session_id` приемлем - каждый запрос возвращает 1 документ |
| Слияние корзин | Операция читает 2 документа и обновляет 2 - scatter-gather на 2 шарда, что допустимо |

**Почему не другие варианты:**
- `user_id` hashed - гостевые корзины с `user_id: null` все попадают на один шард → хотспот
- `session_id` hashed - авторизованные корзины с `session_id: null` создают аналогичный хотспот
- `{ user_id, session_id }` - составной ключ с null-значениями непредсказуем в распределении

### Команды MongoDB

```javascript
// Создать хешированный индекс
db.carts.createIndex({ _id: "hashed" })

// Зашардировать коллекцию
sh.shardCollection("somedb.carts", { _id: "hashed" })
```
