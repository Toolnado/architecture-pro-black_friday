# Задание 9. Настройка чтения с реплик и консистентность

## Контекст

MongoDB replica set: один **primary** (принимает все записи) и два **secondary**
(реплицируют данные асинхронно). Чтение с secondary снижает нагрузку на primary,
но данные могут быть устаревшими на время задержки репликации (replication lag).

Read preference определяет, с какого узла читать:

| Read Preference | Поведение |
|---|---|
| `primary` | Только с primary - всегда свежие данные |
| `primaryPreferred` | Primary, при недоступности - secondary |
| `secondary` | Только с secondary - возможна задержка |
| `secondaryPreferred` | Secondary, при недоступности - primary |
| `nearest` | С ближайшего узла по latency |

## 1. Коллекция `products`

### Таблица операций

| Операция | Read Preference | Read Concern | Обоснование |
|---|---|---|---|
| Проверка остатка при оформлении заказа | `primary` | `linearizable` | Риск overselling: продать товар с нулевым остатком. Нельзя читать устаревшие данные |
| Отображение остатка на странице товара | `secondaryPreferred` | `local` | Показ «50 вместо 48» не критичен - пользователь всё равно не покупает в момент просмотра |
| Страница товара (описание, атрибуты) | `secondary` | `local` | Редко меняется, высокая нагрузка на чтение, задержка незаметна |
| Листинг категории + фильтр по цене | `secondary` | `local` | Каталог меняется редко, поисковая нагрузка высокая - secondary разгружает primary |
| Обновление остатков при покупке | запись → `primary` | `majority` | Запись всегда идёт на primary; majority гарантирует, что данные не потеряются при failover |

### Допустимая задержка репликации

| Операция | Допустимый lag |
|---|---|
| Страница товара, листинг | до 60 секунд |
| Отображение остатка | до 10 секунд |
| Проверка остатка при покупке | 0 (только primary) |

## 2. Коллекция `orders`

### Таблица операций

| Операция | Read Preference | Read Concern | Обоснование |
|---|---|---|---|
| История заказов пользователя | `secondary` | `local` | Пользователь смотрит прошлые заказы - задержка 1–2 с незаметна |
| Отображение статуса заказа (общий случай) | `secondaryPreferred` | `local` | Статус «доставлен / в пути» меняется нечасто, небольшая задержка приемлема |
| Статус заказа сразу после оформления | `primary` | `majority` | Пользователь только что создал заказ и ожидает увидеть «принят» - нельзя вернуть пустой результат из-за lag |
| Создание заказа | запись → `primary` | `majority` | Критичная запись; majority предотвращает потерю при failover |
| Аналитика / отчёты | `secondary` | `local` | Аналитические запросы тяжёлые - secondary изолирует от production-трафика |

### Допустимая задержка репликации

| Операция | Допустимый lag |
|---|---|
| История заказов | до 30 секунд |
| Статус заказа (общий случай) | до 10 секунд |
| Статус сразу после оформления | 0 (primary) или causal consistency |
| Аналитика | до 5 минут |

### Causal Consistency - статус сразу после создания заказа

Вместо жёсткой привязки к primary можно использовать **causal consistency session**:
MongoDB гарантирует, что последующее чтение увидит результат предыдущей записи
в той же сессии, даже если оно идёт на secondary.

## 3. Коллекция `carts`

### Таблица операций

| Операция | Read Preference | Read Concern | Обоснование |
|---|---|---|---|
| Получение активной корзины (общий случай) | `primaryPreferred` | `local` | Корзина меняется часто; устаревшие данные могут показать удалённый товар или неверное количество |
| Чтение корзины после добавления товара | `primary` | `majority` | Read-after-write: пользователь должен видеть свой только что добавленный товар |
| Слияние гостевой корзины в пользовательскую | `primary` | `majority` | Read-modify-write атомарная операция; устаревшие данные гостевой корзины приведут к потере товаров |
| Отметить корзину как ordered | запись → `primary` | `majority` | Финальная смена статуса; нельзя потерять при failover |
| Отображение брошенных корзин (аналитика) | `secondary` | `local` | Не влияет на бизнес-логику |

### Допустимая задержка репликации

| Операция | Допустимый lag |
|---|---|
| Получение активной корзины | до 2 секунд |
| После добавления / удаления товара | 0 (primary) |
| Слияние корзин | 0 (primary) |
| Аналитика брошенных корзин | до 10 минут |

## 5. Ключевые принципы выбора

**Secondary можно использовать, если:**
- Данные меняются редко (каталог, описания товаров)
- Задержка в отображении не влияет на бизнес-решения (история заказов, статус в общем случае)
- Операция аналитическая и не блокирует пользователя

**Primary обязателен, если:**
- Операция участвует в транзакции (остатки, создание заказа)
- После записи немедленно следует чтение (добавление в корзину → показ корзины)
- Ошибка из-за устаревших данных приводит к потере денег или товара (overselling, потеря товаров при слиянии корзин)
- Read-modify-write операция (слияние корзин)
